iOS 播放和录制音频

Posted on 2018-09-25 | In iOS开发
Words count in article: 863 字 | Reading time ≈ 3 分钟

AV Foundation最开始是一个仅针对音频的框架，在iOS3.0中，增加了音频录制的功能，现在我们通过学习AVAudioPlayer和AVAudioRecoder类来实现应用程序中添加音频播放和音频录制的功能。

#####理解和使用音频会话

音频会话简单的理解就是可以在我们使用音乐播放的过程中，收到电话通知，这时候就需要有一个管理的处理来将音乐播放停止，开始播放来电铃声。

所有的iOS应用程序都具有音频会话，无论其是否使用。AVFoundation定义了七种分类来描述应用程序锁使用的音频行为

  • Ambient 游戏、效率应用程序
  • Solo Ambient 游戏、效率应用程序
  • Playback 音频和视频播放器
  • Record 录音机、音频捕捉
  • Play and Record VoIP、语音聊天
  • Audio Processing 离线会话和处理
  • Multi-Route 使用外部硬件的高级A/V应用程序

其中，来实现播放音频的方法很简单，将本地要播放的文件放入我们的项目中，新建一个播放器就可以轻松实现音频的播放的功能了。

我们也可以自己修改音乐播放器的音量，这个是独立于系统的音量，还有一个需要我们注意的就是当音频被打断的时候的处理，我们需要监听线路的变化，并作出相应的处理。相关的方法是AVAudioSessionRoute…相关的类，这里大家可以自行查询。

#####使用AVAudioRecorder⏺音频

下面我们看一下如何来录制音频，首先我们还是需要导入#import <AVFoundation/AVFoundation.h>框架，录制的代码很简单,主要使用到的类是AVAudioRecorder

  • AVFormatIDKey键定义了写入内容的音频格式，这里我们设置支持的格式是kAudioFormatAppleIMA4，这种格式的压缩格式会显著缩小文件，还能保证高质量的音频内容
  • AVSampleRateKey键定义了录音器的采样率。采样率定义了对输入的模拟音频信号每一秒内的采样率。使用低的采样率，回导致粗粒度、AM广播累心给的录制效果，不过文件会比较小，使用高的采样率文件就会较大。
  • AVNumberOfChannelsKey键用来定义记录音频通道内的通道数，指定默认值1意味着使用单声道录制，设置为2意味着立体声录制，如果不是使用外部硬件进行录制，通常使用单声道录制。当我们停止录音的时候，可以调用saveRecordingWithName:completionHandler:方法，这个方法通过唯一文件名将tmp目录中的录制文件复制到Documents目录，如果复制操作成功，会调用完成块，传递会一个新的包含名字和录制音频URL。

iOS 中的内存管理

Posted on 2018-06-12 | In iOS开发
Words count in article: 1,298 字 | Reading time ≈ 5 分钟

iOS中的内存管理

内存是我们平常生活中用来存储数据的存在，软件在运行的时候会分配和使用设备的内存资源，因此需要我们在软件开发的过程中，进行合理有效的内存管理，以保证快速高效的内存分配，提升软件的优越性。

首先我们来讨论一下内存中的分区：

1.内存中分为五大区域

在这里我们只做简单的介绍

栈区：存储局部变量的区域，当其作用域执行完毕之后，就会被系统收回，是系统自动管理的。

堆区： 存储OC的对象，手动申请的字节空间，需要自己释放

BSS段（Block Started by Symbol）：未初始化的全局变量和静态变量，一旦初始化就会从BSS段中回收掉，转存到数据段中

数据段： 存储已经初始化的全局变量和静态变量，以及常量数据，直到程序结束时才会被立即收回。数据段属于静态内存分配。

代码段： 通常是指用来存放程序执行代码的一块内存区域，这部分区域的大小在程序运行前就已经确定了，并且内存区域通常属于只读。在代码段中，也有可能包含一些只读的常量数据。

除了堆区，其他区域中存储的数据，都是由系统自动释放的。堆区中存放的OC的对象，是不会自动释放的，如果不主动释放，那么将在程序结束的时候释放。

内存的管理原则

就是我们常说的引用计数的问题，OC的对象都继承于NSObject，该对象有一个属性，叫做retainCount,也就是我们说的引用计数器，类型为unsigned long，占据八个字节，每一个对象负责维护对象所引用的计数值，当一个新的引用指向对象，那么这个对象的引用计数会增加1，当我们新创建一个对象的时候这个对象的引用计数器的值默认为1，当这个对象被少一次引用的时候这个对象的引用计数值减1，当这个对象的引用计数值为0的时候就说明了这个对象没有被使用，这个时候系统会自动回收掉此对象。

内存管理的方式：

1.MRC: Manual Reference Counting 手动引用计数器，需要我们手动管理对象引用计数的值。

2.ARC： Automatic Reference Counting 自动引用计数，系统自动的改变对象引用计数的值（iOS5之后），ARC是基于MRC的

这里我们重点讨论一下ARC：

ARC是automatic reference counting 自动引用计数，在程序编译时自动加入retain/release。 在对象被创建时retaincount + 1,在对象被release时count -1 ，当count=0时，对象销毁。程序中加入autoreleasepool对象会由系统自动加上autorelease方法，如果该对象引用计数为0，则销毁。ARC是为了解决MRC手动管理内存存在的一些问题而诞生的。

MRC下内存管理的缺点：

  • 释放一个堆内存时，首先要确定指向这个堆空间的指针都被release了。（要避免提前释放）
  • 释放指针指向的堆空间，首先要确定哪些指向同一个堆，这些指针只能释放一次。（避免释放多次，造成内存泄漏）
  • 模块化操作时，对象可能被多个模块创建和使用，不能确定最后由谁来进行释放。
  • 多线程操作时，不确定那个线程最后使用完毕。

但是ARC下也可能会出现内存泄漏的情况，比如如下的情况：

  • block的循环引用，可以将引用的一方变成weak，来避免循环引用。
  • NSTimer 在使用NSTimer addTarget时，为了防止target被释放而导致的程序异常，timer会持有target，这个也是一处内存泄漏的隐患。
  • 非OBJC对象 ARC是自动检测objc对象的，如果不是objc对象就无能为力了，C语言使用malloc开辟，free释放 c++使用new开辟，delete释放，但是在ARC下，不会添加非objc对象释放语句，如果没有去释放，也会造成内存泄漏。

每一个工程中都有一个main.m文件

int main(int argc, char * argv[]) {

@autoreleasepool {

return UIApplicationMain(argc, argv, nil, NSStringFromClass([AppDelegate class]));

}

}

autoreleasepool是用来做autorelease变量释放的，autorealease不会马上释放，当他到了最近的pool release时会检查retain count是不是为0，为0就会释放。

自动释放池的底层实现：

自动释放池以栈的形式实现： 当你创建一个新的自动释放池时，他将被添加到栈顶，当一个对象收到发送autorelease消息时，他被添加到当前线程的处于栈顶的自动释放池中，当自动释放池被回收时，他们从栈中被删除，并且会给池子里所有的对象都会做一次release操作。

React Native 简介

  • React Native是什么？（一下简称RN）
    RN是Facebook在2015年4月开源的跨平台的移动应用开发框架，是Facebook早先开源的JS框架React在原生移动应用平台的衍生产物，目前支持iOS和安卓两大平台。RN使用JavaScript语言，类似HTML的JSX，以及CSS来开发移动应用，因此熟悉Web前端开发的技术人员很容易进入到移动应用开发领域。 RN使你能够在JavaScript和React的基础上获得完全一致的开发体验。

    Native指的就是原生的APP，也就是用原生开发方式开发出来的应用.
    React是一个用于构建用户界面的js库，React是声明式的，高效率的，并兼具灵活性，其中用于构建用户界面的js库有很多，如angular，vue，jQuery，ext等，都可以称为用户构建用户界面的js库。但是React与它们又有所不同，下面我们来分别介绍一下React的这几个特性：

    1.声明式：声明式是由JSX语法来表现的

    JSX是使XML可以嵌入JavaScript的语法，可以更加直观的描述UI，JSX是JavaScript对象，可以理解为createElement的语法糖。

这里的声明式指的就是上面代码中用大括号括起来的部分{this.props.name}，这里类似于angular中的数据绑定，声明了UI中的表现与js中数据的关系。我们可以吧任意合法的JavaScript表达式通过括号嵌入到JSX语句中，括号的意思是括号内部为一个JS变量或者表达式，需要执行后取值。

**2.高效率：高效率是由虚拟的DOM节点来实现的**

虚拟DOM技术指的是JavaScript Object模拟DOM树，计算变更，减少重绘，提升效率的一种技术实现。

**3.灵活性**

这个是一个很广的范围，主要有安装的灵活性和编写代码的灵活性等，其实就是components即**组件化**

RN通过components来进行组件化。他把UI分割成独立，可重用的部件。React有它自己的组件化方式，就是所有模块都继承自React.Components,这个要比angular的modules更加的轻量级，结构也更加的简单

介绍完了他的特点，下面我们来总结一下：

React通过虚拟的DOM实现了对UI层的解耦，然后将浏览器的DOM树替换成Native的UI模块，使React在移动端进行渲染，最后采用Native的核心API完成React与Native的通信，由于采用Native原生UI作为展现层，所以RN具有可以媲美原生应用的优秀用户体验和性能优势。RN是基于React技术编写Native应用的一门技术，他继承了React简明、高效、灵活的特点，又具有Native优秀的用户体验。

RN采用JS编码，编写的应用具有跨平台，可以热更新的优势。

KVO的内部实现原理

Posted on 2018-05-19 | In iOS开发
Words count in article: 845 字 | Reading time ≈ 3 分钟

1.KVO是什么？

KVO，就是Key-Value Observing的简称(键值观察),是iOS设计模式中的一种，当我们设置了观察某个对象的某个属性，当这个属性发生改变的时候，这个对象会收到通知，然后执行进一步的操作。

2.我们怎么使用KVO？

我们创建了一个Person类，在这个类中有一个name的属性，我们设置监测这个值的改变，当我们点击屏幕的时候，就改变name的值，我们看到打印的内容,提示我们监测的值发生了改变，这个就是最简单的KVO的应用。

在我们实际开发项目的过程中，可能用到的要比这个复杂的多，这个时候我们可以使用RAC来进行监测，Swift的也有相应的大神贡献的库，使用起来简直不要太爽。当然这里使用KVO的时候，不要忘了在我们不需要监测的时候，移除监测者。

3.KVO 的实现原理

经过了上面的两步，我们也已经最KVO的概念和基本的应用有了基础的了解了，那么下面就到了我们今天要讨论的重点了，这个KVO使用起来这么的方便，那他底层到底是怎么实现的呢。

  • 其实这个是通过我们的强大的Runtime（运行时）来进行实现的。当你第一次观察某个对象时，系统就会在运行时动态的创建一个该类的派生类,在这个派生类中重写基类中任何被观察属性的setter方法，派生类在被重写的setter方法内实现真正的通知机制。

  • 在我们的例子中，我们创建的类是Person类，那么将会生成的派生类类名为NSKVONotifying_Person，每一个类对象中都有一个isa指针指向当前类，当一个类对象第一次被观察的时候，系统会将isa指针指向动态生成的派生类，从而在给被监控属性赋值的时候执行的是派生类的setter方法。

  • 键值观察通知依赖NSObject的两个方法：willChangeValueForKey:和didChangevlueForKey: ;
    在一个被观察的属性发生改变之前，willChangeValueForKey方法一定会被调用，这个时候回记录改变之前的的旧值，当这个值发生改变之后会调用didChangevlueForKey方法，记录新的值，然后observeValueForKey:ofObject:change:context:方法也会被调用。

  • 在KVO的这个实现机制中苹果还在内部重写了Class方法，让我们误认为我们使用的还是当前的类，隐藏生成的派生类。最后盗一张图，详细展示一下KVO实现（侵必删！！！）
